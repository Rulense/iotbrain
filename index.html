WHY

IoT knowledge is scattered and perishable.

How to deploy a model with TensorRT on a Jetson Orin. Which kernel and driver combination gets a CSI camera working on a Raspberry Pi 5. How to do OTA updates on an ESP32 fleet without bricking it. How to cross-compile a library for an ARM target. What breaks along the way, and how to get past it.

All of it exists — spread across vendor forums, sample repos, release notes, and the hard-won experience of engineers. None of it is in one place an agent, or a human, can reliably draw on. So every team relearns its devices from scratch.

What the brain holds

5 ENTRY TYPES · PLAIN MARKDOWN

Not a troubleshooting database — a general knowledge store covering the whole IoT development lifecycle, scoped per device and platform.

recipe
A verified way to accomplish a task
Deploy YOLO with TensorRT on a Jetson Orin Nano, end to end
config
A known-working configuration
The exact wheel + runtime combo that imports cleanly on a given OS image
matrix
Version compatibility knowledge
SDK × driver × framework support across a platform’s releases
gotcha
A trap to avoid proactively
A camera driver quirk that silently drops frames on one firmware release
fix
An error and its verified solution
A shared-library import failure → the exact resolution, per platform

The skills put it to work: a main iot-dev skill consults the brain for any task, the brain-distill skill grows it, and domain packs for the major workflows arrive in v0.2.

The loop

FOR EVERY TASK, NOT JUST BROKEN ONES

Build, deploy, integrate, optimize, debug — every task starts by consulting the brain and can end by growing it.

01 CONSULT
Any IoT task begins with a device- and version-aware grep of the brain.
02 APPLY
Brain knows: apply known recipes and configs, step around gotchas before hitting them.
03 FIGURE OUT
Brain is silent: research the internet, experiment, and verify on real hardware.
04 DISTILL
Learned something new and verified? The distiller writes the entry and drafts a community PR.

A working vision pipeline produces a recipe. A clean build combo produces a config. A lost hour produces a gotcha. Debugging is just one way the brain grows.

How it works

FOUR DESIGN BETS
NO INFRASTRUCTURE
Plain markdown in the repo, retrieved by grep. Exact error strings, package names, and device models are the search keys — what agents search best. No vector database, no Docker, no index to go stale.
DEVICE- & VERSION-SCOPED
Every entry is pinned to device models and platform/SDK versions. The agent checks applicability before trusting knowledge, so nothing stale — or from the wrong device — gets confidently misapplied.
HUMAN-REVIEWED GROWTH
The only write path into the shared brain is a pull request. The distiller formats entries; the community reviews truth. Nothing is published without the contributing user approving the exact content first.
TRANSPARENT VERIFICATION
Every entry declares its status: verified means it worked on real hardware, or matches current official docs — with the citation and check date recorded. Everything else is honestly labeled unverified, so the agent knows exactly how much to trust it.

What’s inside

skills/ is how the agent works — the companion, the distiller, and the domain packs arriving in v0.2. brain/ is what it knows — organized by domain, mapped by a one-line-per-entry INDEX.

Entries carry a company field today — nvidia first. Knowledge that can’t go public lives in the ~/.iotbrain/local/ overlay on your machine, consulted right beside the shared brain. Platform trees are the growth path: as entries land from more vendors, the brain splits by platform — so a Jetson entry never pollutes a Pi session.

iotbrain/ ├── skills/ │ ├── iot-dev/ brain consultation, any task │ ├── brain-distill/ learnings → entries + PRs │ ├── … vendored NVIDIA device skills arriving │ ├── vision-pipeline/ cameras, capture, inference (v0.2) │ ├── iot-connect/ MQTT, cloud, fleet deployment (v0.2) │ └── sdk-build/ cross-compiling, packaging (v0.2) └── brain/ ├── INDEX.md one line per entry ├── setup/ flashing, recovery, first boot ├── ml-stack/ wheels, CUDA, TensorRT … ├── vision/ CSI cameras, Argus, GStreamer … ├── iot/ MQTT, OTA, fleet networking … ├── sdk-dev/ cross-compiling, linking … └── runtime/ power, thermals, memory … (entries are company-scoped today — platform trees come with multi-vendor growth)
